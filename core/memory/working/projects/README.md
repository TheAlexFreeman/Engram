# Projects

`memory/working/projects/` holds one subfolder per active initiative. The auto-generated `SUMMARY.md` at this level is the navigator across all projects; each subfolder's own `SUMMARY.md` is the per-project briefing and the entry point a cold-starting agent should read after the navigator.

**Layout at this level.**

- `SUMMARY.md` — auto-generated projects navigator. Regenerated by `memory_generate_summary`; can go stale between regenerations.
- `OUT/` — emitted decisions and artifacts that cross project boundaries.
- `ACCESS.jsonl` — append-only access audit for this folder.
- `<project>/` — one folder per project (e.g. `rate-my-set/`, `skills-expansion/`).

**Per-project substructure.** Each `<project>/` folder carries the same skeleton: `SUMMARY.md` (agent-generated briefing), `IN/` (staged external material), `OUT/` (emitted artifacts scoped to this project), `docs/` (reference material), `plans/` (YAML plans managed by the `memory_plan_*` tools), `notes/` (working notes), `questions.md` (open questions), and `operations.jsonl` (append-only operations audit).

**Cold-start entry point.** From `CLAUDE.md`: call `memory_session_bootstrap`, read this folder's `SUMMARY.md` and the per-project `SUMMARY.md`, then call `memory_context_project(project=<name>)` for the enriched bundle (or `memory_context_project_lite` when guaranteed sub-500ms latency matters).

The full lifecycle doc (who writes into each subfolder, when material is promoted upstream, how stale snapshots are reviewed) lives in the P1-C expansion of this README. Until that lands, the definitions above plus the per-project `SUMMARY.md` files are authoritative.
