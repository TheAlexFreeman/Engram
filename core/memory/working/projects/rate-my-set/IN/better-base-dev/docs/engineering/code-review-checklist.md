# Code Review Checklist

## Review Order

Use this order so high-impact issues are surfaced first.

1. Correctness and regressions.
2. Security and data safety.
3. Performance and reliability.
4. Tests and observability.
5. Maintainability and style.

## Correctness

- Does the change do what the task says, including edge cases?
- Are defaults and fallbacks explicit and safe?
- Are API contracts preserved for existing callers?
- Are migrations, schema changes, and generated artifacts included when required?
- Were migration files generated by tooling (`makemigrations`, etc.) rather than
  manually authored or edited?
- For changes in `backend/**/ops.py` or `backend/**/ops/**/*.py`, is business logic kept
  in operations (not duplicated in views/tasks/commands), with explicit action-oriented
  operation boundaries?

## Security and Data Safety

- Is input validation present where trust boundaries are crossed?
- Are secrets or tokens excluded from logs, commits, and generated files?
- Are file/path operations constrained to expected directories?
- For auth/permissions logic, is the failure mode deny-by-default?

## Performance and Reliability

- Any obvious n+1 queries, unnecessary rerenders, or repeated expensive work?
- Any blocking operations on critical request/render paths that should be deferred?
- Are retries/timeouts/backoff handled where network or subprocess calls are
  introduced?

## Tests and Verification

- Are meaningful tests added or updated for changed behavior?
- For model changes, are there dedicated `repr(...)` and `str(...)` tests (not only
  incidental coverage)?
- Are brittle or over-mocked tests avoided?
- Were relevant lint/type/test commands run for touched areas?
- If checks were skipped, is the gap called out explicitly?

## Maintainability

- Is the implementation consistent with local patterns in this repository?
- Are names clear and intent-revealing?
- Is dead code removed, and are comments/docstrings still accurate?
- Are docs updated when user-facing behavior, commands, or configuration changed?

## Review Output Template

Use a clear, severity-first format:

1. Findings (highest severity first, with file references).
2. Open questions or assumptions.
3. Brief summary of what changed and what was validated.
